The lines behind this — staged entrance

Four declarations: a transition, a stagger, and where the element starts. Durations multiply --wel-motion, so the reduced-motion preference (or the switch below) makes them instant.

.rise {
  transition:
    opacity calc(var(--wel-motion-duration-4) * var(--wel-motion)) var(--wel-motion-ease),
    translate calc(var(--wel-motion-duration-4) * var(--wel-motion)) var(--wel-motion-ease);
  transition-delay: calc(var(--rise-order, 0) * 90ms * var(--wel-motion));
  @starting-style { opacity: 0; translate: 0 1.25rem; }
}

Motion is a token, tuned like a colour

Hover any button on this page — it lifts, overshoots, and settles. The lift is one line; the spring and its timing are theme tokens, set the same way this page sets its colours. Welkin components never hard-code a duration — everything says calc(token × --wel-motion) — so retuning three tokens changes the whole site's character:

The lines behind this — the page's motion theme 
:root, [data-theme] {
  --wel-motion-ease: var(--wel-motion-ease-spring);  /* dialogs, toasts, tabs, hovers */
  --wel-motion-duration-1: 180ms;                    /* room for the spring to read */
  --wel-motion-duration-2: 250ms;
}
/* the lift itself — the button already transitions `translate`,
   so the themed spring is what animates it */
.button:hover { translate: 0 calc(-3px * var(--wel-motion)); }

The lines behind this — cards that morph when you filter

Each card owns a view-transition-name; the filter runs inside document.startViewTransition when the browser has it, and as an instant swap when it doesn't (same function either way). The morph's duration rides the same --wel-motion multiplier as everything else.

.work-card { view-transition-name: var(--vt); view-transition-class: folio; }
::view-transition-group(.folio) {
  animation-duration: calc(var(--wel-motion-duration-3) * var(--wel-motion));
  animation-timing-function: var(--wel-motion-ease);
}

const apply = () => { /* set [hidden] per card */ };
button.addEventListener('click', () =>
  document.startViewTransition ? document.startViewTransition(apply) : apply());

The animations you don't write

Dialog fade-and-scale, toast slide-in, tab-indicator morph — these ship with the components. Added CSS on this page for them: none.

Discover

Two weeks of watching how the thing actually gets used. Motion that isn't anchored in a real pause, a real wait, or a real success is decoration — we cut it before it's drawn.

Design

Storyboards first, easing curves second, tooling last. Every scene gets a one-line intent ("reassure", "orient", "celebrate") that survives into the handoff file.

Deliver

Specs as tokens: durations, easings, and choreography shipped as custom properties your engineers drop into the codebase — the same way this page does it.

All of it rides one switch

Every duration on this page — entrances, springs, morphs, the components' own animations — multiplies --wel-motion. Drag it to zero and the page goes still; the OS-level reduced-motion preference does exactly this automatically, with no media queries written here.

Set to 0, then reopen the dialog or re-filter the work grid — instant, not broken. Zero-duration transitions still fire their events, so nothing waits on an animation that never runs.
