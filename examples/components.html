Container queries change the question

Components should respond to the space they are given, not the viewport they happen to be on. A card in a 300px sidebar and the same card in a 900px column each lay themselves out correctly — see the layout doctrine.

Select

Native <select>, styled trigger, UA picker — color-scheme keeps the dropdown dark in dark mode. The caret is gradient-drawn, so it consumes --wel-select-caret-color in both schemes. No JS, ever.

Form controls & validation

Native constraint validation is the engine: errors ship in markup (wired into aria-describedby) and appear via :user-invalid — after interaction, never at page load. [aria-invalid="true"] is the server-rendered twin. Try submitting empty.

Tooltip

Popover-API toggletip in Core: click / Enter / Space toggles, Esc and light dismiss close, and the text is the trigger's aria-describedby description — exposed to AT without ever opening it. Under Enhanced it tethers to the trigger (anchor positioning) and fades. Hover reveal arrives with wel-tooltip (Phase 5).

Carousel

Scroll-snap rail: the browser supplies touch, momentum, keyboard, and RTL — focus the list and use arrow keys. No autoplay, ever (scope decision, 2.2.2). Under Enhanced a scroll-progress bar tracks position; generated markers/buttons join when ::scroll-marker passes intake.

Dialog

Native <dialog>: showModal() gives the focus trap, background inert, Esc, and top layer for free; closing is declarative via form method="dialog". Opening is the one gap — the 3-line script at the end of this page is the whole documented Core path until Invoker Commands graduate.

Popover / Dropdown menu

Fully declarative: popovertarget toggles, light dismiss and Esc close, the invoker's expanded state reaches AT natively — zero JS. Plain lists, no role="menu" (that role promises APG keyboard support the Popover API doesn't supply). Tethered under Enhanced; a designed centred panel in Core.

Accordion

<details>/<summary>: semantics, keyboard, and expanded state are the platform's; exclusivity is one attribute (name) that degrades to independent items where unsupported. Zero accordion JS, ever. Height is instant (interpolate-size awaits intake); the panel fades in under Enhanced.

Exclusive FAQ group (default variant)

Can I self-host?

Yes — the toolkit is a static CSS file plus optional ESM modules; there is no build step or runtime to operate.

Which browsers are supported?

Core works on Baseline Widely Available; Enhanced layers on top behind @supports — see the browser support policy.

Why is opening one closing the other?

The shared name attribute — remove it for independent multi-open items (the default recommendation).
